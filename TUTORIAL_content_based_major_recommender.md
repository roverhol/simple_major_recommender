# Tutorial: Building a Content-Based Major Recommender

This tutorial walks you through how I built a very simple major recommender system using only data from our catalog. The system works by analyzing the text descriptions of courses within each major, calculating how similar majors are based on their vocabulary, grouping similar majors together (clustering), and finally allowing a user to interactively explore these groups to find majors that match their interests.

I used a content-based filtering approach, where the ``content'' is the text describing the courses offered within each major. Following the best practice of starting with simplicity, this tutorial focuses on building recommender using a simple, foundational technique using word frequencies and hierarchical clustering.

## Prerequisites

## Step 1: Data acquisition (preparing my data)

This recommender relies on text descriptions of courses associated with each major. This step is crucial: know your data! The quality and nature of the inputs directly impact the recommendations.

1. **Gather data:** I obtained course descriptions for each major by web scraping Cal Poly Humboldt’s course catalog.  
   - *(Self-contained step)* Use `webscrap.py` or `plo_scraper.py` (adapted for course descriptions) to generate the CSV files.

2. **Format data:** Organize the data into separate CSV files, one for each major. Each CSV should contain at least one column (named `description` by default; see script configuration) that holds the text descriptions of the courses for that major.

3. **Place data:** Put all these CSV files into the `course_descriptions_auto` directory.

4. **Explore (know your data):** Briefly examine the descriptions. What I am asking myself when I see them: Are they detailed? Do they contain jargon specific to certain fields? Do you think there might be biases in how different majors are described? What are common words that might be unrelated to the content of a major? Are the courses all required or are options included? Understanding this helps interpret and evaluate the results later.

Example `biology_descriptions.csv`:

```csv
course_code,title,description
BIO101,Intro Biology,"Fundamentals of biology, covering cell structure, genetics, evolution..."
BIO205,Ecology,"Study of interactions between organisms and their environment..."
...
```

*(Note: Only the `description` column is strictly required by the frequency calculation script.)*

## Step 2: Text preprocessing and feature extraction (word frequencies)

Convert the raw text descriptions into a numerical format you can use for comparison: word frequency counts. This is a key part of feature engineering in the pipeline.

- **Purpose:** The `calculate_word_frequencies.py` script reads each major’s description CSV, cleans the text (lowercase, removes punctuation and common “stopwords” like “the”, “a”, “is”), and counts how many times each meaningful word appears.

- **Run the script:** Open the VS Code terminal (**Terminal → New Terminal**), ensure your working directory is `your_project_directory`, and run:

  ```bash
  python code/calculate_word_frequencies.py
  ```

- **Output:** The script creates a directory `word_frequencies`. Inside, you get a CSV per major listing words and their counts (frequencies). These frequencies are the “features” of each major.

Example `word_frequencies/biology_descriptions_frequencies.csv`:

```csv
Word,Frequency
biology,15
cell,12
genetics,10
evolution,9
organisms,8
environment,7
...
```

## Step 3: (Optional) Visualize word clouds

Word clouds give a quick visual summary of the most frequent words in each major’s descriptions—another way to know your data and check text processing.

- **Purpose:** The `generate_wordclouds.py` script reads the original description CSVs, processes text similarly to the frequency script, and generates a word cloud image for each major.

- **Run the script:**

  ```bash
  python code/generate_wordclouds.py
  ```

- **Output:** Creates the `word_clouds` directory with PNG images. Larger words in the cloud appeared more frequently in that major’s descriptions.

## Step 4: Calculate similarity and cluster majors (model training)

With word frequencies computed, measure how similar majors are and group them. This step is the model-training phase of the simple recommender.

**Purpose:** The `cluster_majors.py` script:

- Loads word frequency data from `word_frequencies`.
- Calculates a similarity score (using the **Sørensen–Dice** coefficient: remind this is like Jaccard but with average size instead of union in the denominator) for every pair of majors from shared vocabulary and word counts.
- Saves pairwise similarity to `major_similarity_matrix.csv`.
- Converts similarity scores to distances (required for clustering).
- Performs **hierarchical clustering** using the `ward` method (configurable) to build a tree (**dendrogram**).
- Saves the linkage matrix to `major_linkage.pkl`.
- Chooses a distance threshold to cut the dendrogram (aiming for clusters that are not too large; configurable via `MAX_CLUSTER_SIZE`) and writes `optimal_threshold.txt`.
- Saves a dendrogram figure as `major_clusters_dendrogram.png`.
- Assigns each major to a cluster at that threshold and saves `major_cluster_assignments.csv`.

**Run the script:**

```bash
python code/cluster_majors.py
```

**Key outputs:**

- **`major_clusters_dendrogram.png`:** Open in VS Code or your file browser. Majors on nearby branches are similar by course-description vocabulary. A red dashed line (if present) shows where the tree was cut to form final clusters—useful for evaluating clustering quality.
- **`major_cluster_assignments.csv`:** Each major and its cluster ID; same ID means grouped together.
- **`major_linkage.pkl`** and **`optimal_threshold.txt`:** Needed for the interactive step.

## Step 5: Interactive exploration (simple serving)

A user can explore clusters interactively to find majors related to their interests. This is a basic form of model serving.

- **Purpose:** The `explore_clusters_interactive.py` script uses saved clustering (`major_linkage.pkl`) and word frequencies to walk a user down the dendrogram. At each split it shows representative words that distinguish the two sub-branches and asks the user to pick the side that sounds more interesting, until a reasonably small cluster (from `optimal_threshold.txt`) is reached.

- **Run the script:**

  ```bash
  python code/explore_clusters_interactive.py
  ```

- **Interaction:** The script prints to the VS Code terminal, for example:

  ```text
  Split at distance: 0.8500
  --------------------
  Option A focuses on concepts like:

    math | calculus | algebra | proof | logic | theorem | analysis

  Option B focuses on concepts like:

    data | programming | algorithm | software | network | database | system

  --------------------
  Which option sounds more interesting? (A/B):
  ```

  Enter **A** or **B** based on your interest. The script continues until it prints the list of majors in the cluster you navigated to.

## Conclusion and summary of best practices

You have just travelled through the full pipeline from data acquisition to a simple interactive system. This project is gives us the technical foundation for understanding how recommendation systems can be built with content analysis and unsupervised learning.

**Reflection:**
There is a lot of code involved to build this simple recommender.  Here's what I hope you remember from this tutorial:

- **Start with simplicity:** Word frequencies and hierarchical clustering are a good baseline before adding more sophisticated approaches such as TF–IDF, embeddings, or other clustering algorithms. Establish evaluation criteria early.
- **Know your data:** Course-description quality matters. Biased, incomplete, or overly generic descriptions yield weaker clusters. Did word clouds or frequency lists any of the majors surprise you?  Also notice that I scraped all courses associated with a major, some of these are required and some may be options within a concentration. 
- **Consider the full pipeline:** Data collection (Step 1), preprocessing and features (Step 2), training (Step 4), evaluation (dendrogram and clusters), and serving (Step 5).
- **Balance multiple objectives:** This model mainly optimizes textual similarity. A real system might also balance diversity, popularity, or career outcomes. Interactive exploration gives the user some control.

**Ethical considerations to think about for future work:**

- **Bias amplification:** Can catalog language reinforce stereotypes about fields? A recommender might amplify those biases.
- **Filter bubbles:** Could the system steer students only toward majors they already know, limiting discovery of less obvious fits?
- **Transparency:** The interactive explorer shows distinguishing words, is more a more transparent method better than using a black box method. Is transparency of the recommendations important in this context? Should you be able to explain why majors cluster together?
- **Privacy:** This walkthrough uses public course data. If real student data were included, privacy would need explicit safeguards.
